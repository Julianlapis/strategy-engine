# The Pipeline

A unified workflow from written input to shipped product.
Designed for Julian Alexander. Operates through `/strategy-engine` and `/arc`.

---

## Why Two Systems

Arc goes from idea to shipped code. It's built for solo builders who want to start building.
Strategy-engine adds the strategic layer that Arc doesn't have: research, distillation,
briefs, specs, and component maps at enterprise and agency quality. For work that needs
strategic rigor before a single line of code gets written, strategy-engine produces the
inputs that make Arc's build phase sharper.

**You can use Arc without strategy-engine.** For small personal projects where you already
know what you're building, go straight to `/arc:vision` or `/arc:ideate`. The pipeline
exists for when the problem needs to be understood before it's solved.

---

## The Rule

Every project that runs through the pipeline hits every step. The depth dial changes. The sequence does not.

**Solo build triggers Arc. Team build stays in strategy-engine. Arc:vision is the only bridge between the two systems.** (Teams still use Arc for `vision.md` generation — but only that.)

---

## The Sequence

```
1. INTAKE          What are we doing and why?
2. RESEARCH        What's true about the market, audience, category?
3. DISTILL         What's the tension, insight, way in?
4. BRIEF           The core document. Clear. Unambiguous. Original.
5. SPEC            Capabilities, requirements, what we're building.
6a. MAP            Components, zones, flows — the human-readable structure.
6b. VISION.MD      Agent north star, generated from the brief's handoff contract.
7. BUILD           Design and development.
```

Steps 1-5 produce **one artifact per step** that is the same for humans and agents.
(Step 4 also generates a sidecar file, `04-brief-handoff.yaml`, auto-generated by
`/strategy:product` — but the brief itself is the artifact.)

Step 6 is where the fork happens: 6a is the human-facing map (strategy-engine),
6b is the agent-facing north star (Arc). `.specs/` is also generated at Step 6
from `05-spec.md` — it's the per-capability structured format that agents consume
alongside `vision.md`.

Step 7 is execution.

---

## The Fork

Everything before the MAP is a single document. The strategic brief is the core.
The research dossier feeds the brief. The spec extends the brief into capabilities.
All of these are markdown. All live in the repo. All are readable by both humans and agents.

At Step 6 (MAP), the artifact serves both audiences in a single file:

```
                    BRIEF (single document, same for everyone)
                      |
                    SPEC (single document, capabilities + requirements)
                      |
                    MAP (dual-audience: YAML frontmatter for agents,
                         markdown body for humans, one file per artifact)
                      +
                    docs/vision.md (Arc:vision generates from brief sidecar)
```

**MAP files** = Dual-audience. Each component card, zone map, and user flow lives
in `docs/map/` as a single markdown file with YAML frontmatter (agents parse the
structured data) and markdown body (humans read the descriptions). One source of
truth. No synchronization problem.

**docs/vision.md** = The agent north star. 500-700 words. Arc:vision generates this
from `docs/strategy/04-brief-handoff.yaml` (a sidecar file auto-generated from the
brief). The brief itself stays clean for copy-pasting into decks and Notion.

### The Typed Handoff Contract: Brief → Vision.md

The brief generates a sidecar file (`docs/strategy/04-brief-handoff.yaml`) that
Arc:vision reads to produce `docs/vision.md`. The YAML is NOT embedded in the brief
(it breaks when copy-pasted into decks or Notion). The sidecar is auto-generated
by `/strategy:product` (vision mode) during brief writing and lives alongside it.

Arc:vision checks for this file first. If found, it skips the 5-question Q&A and
generates directly from the contract. If not found, it falls back to Q&A.

Sidecar format (`04-brief-handoff.yaml`):
```yaml
# Auto-generated from 04-brief.md. Do not edit directly.
purpose: One sentence. What this product/service does.
audience: Who it's for. Behavior-based, not demographic.
goals:
  - Goal 1 (measurable)
  - Goal 2 (measurable)
  - Goal 3 (measurable)
success: How we know it's working. Quantitative.
non_goals:
  - What we are NOT building (explicit)
  - What we are NOT optimizing for
principles:
  - Decision-making heuristic 1
  - Decision-making heuristic 2
```

Arc:vision consumes this. Humans ignore it (it's at the bottom, clearly marked).
This prevents the brief and vision.md from diverging.

### What strategy-engine:product needs to learn (NEW)

Strategy-engine:product currently has two modes: vision and spec.
It needs a third mode: **map**.

Map mode takes the spec as input and produces:
- Component cards (Cenex pattern: what it is, where it lives, requirements box)
- Zone maps for key pages (fixed/flexible/optional taxonomy)
- User flow diagrams (task flows with decision points)
- Open questions log

This is the artifact that human designers and developers actually work from.
It's strategy-engine's job because it's stakeholder-facing, not codebase-aware.

### When to use courtroom / pressure-test / narrative

These are **quality amplifiers**, not pipeline steps:

- `/strategy-engine:pressure-test` — Run on the brief at M/L before locking it. Single-agent adversarial critique.
- `/strategy-engine:courtroom` — Run on the brief at L when stakes are high. 10-agent debate. Use sparingly.
- `/strategy-engine:narrative` — When you need slide-ready copy from the brief (for client presentations). Not part of the build pipeline — it's a parallel output for stakeholder communication.
- `/strategy-engine:narrative-review` — Score the narrative before it goes to client. Quality gate.
- `/strategy-engine:distill` — Compress copy. Use before building a deck from the brief.

### Arc skills beyond vision (solo builds only)

- `/arc:ideate` — Feature design with wireframes. 3-act conversation. Expert review.
- `/arc:build` — Small-to-medium features with agent orchestration.
- `/arc:implement` — Full TDD execution against a plan.
- `/arc:design` — UI design with aesthetic direction.
- `/arc:review` — Expert review panel on code/plan.
- `/arc:verify` — Build, typecheck, lint, test.
- `/arc:letsgo` — Production readiness checklist.
- `/arc:harden` — Error states, edge cases, input validation.
- `/arc:polish` — Visual refinement pass.

These compose into Julian's solo build workflow:
```
arc:vision → arc:ideate (per feature) → arc:design → arc:build/implement →
arc:verify → arc:review → arc:harden → arc:polish → arc:letsgo
```

## The Strategic Brief: Template + Scoring Rubric

The brief is the load-bearing document. Everything before it (research, distill) makes
it sharper. Everything after it (spec, map, build) derives from it. When the client
pushes back, when the team gets overwhelmed, when features change, when six months pass
and nobody remembers the kickoff meeting: the brief is what survives.

### The Rallying Cry

Every brief has a core statement at its center. 2-5 words. The thing the entire strategy
collapses to when someone asks "what are we doing?" in a hallway.

**What it is:**
- GamePass: "More gaming"
- A streaming service redesign: "Watch, don't browse"
- A financial dashboard: "Your money, visible"

**What it does:**
- Fits on a sticky note
- A designer can gut-check a layout against it ("does this help them do more gaming?")
- A developer can resolve an edge case against it ("does this make the money more visible?")
- A client can test a feature request against it ("does this help people watch, not browse?")
- Survives six months without needing context

**How to test it:**
- Can someone who wasn't in the room repeat it from memory after hearing it once?
- Does it make a choice? ("More gaming" chose breadth over depth. "Watch, don't browse" chose content surfacing over catalog organization.)
- Would a competitor's strategy use different words? If any brand in the category could claim it, it's not specific enough.

### Brief Structure

```markdown
# [Project Name] Strategic Brief

## The Rallying Cry
[2-5 words. The core statement.]

## The Problem
[Who has this problem? What's broken? Be specific. Use data from the research.
Not "users struggle with X." Name the user. Name the struggle. Name the cost.]

## The Audience
[Behavior-based, not demographic. What they do, not who they are.
"25% of men under 30 already bet online" not "males 18-34."
What they believe. What they're willing to pay for. What they reject.]

## The Proposition
[What we're building and why it wins. One paragraph. Three sentences max.
Sentence 1: What it is. Sentence 2: Why it's different. Sentence 3: Why now.]

## Success Metrics
[3-5 measurable outcomes. Numbers, not vibes.
Each metric has a timeframe and a target.
"1,000 active users within 90 days of launch" not "strong user adoption."]

## Constraints
[What we're working within. Budget. Timeline. Tech stack. Legal.
Organizational politics if relevant. Content availability. Team size.]

## What We're NOT Doing
[The most important section after the rallying cry. Explicit exclusions.
Things the client might expect. Things the team might assume. Things that
feel obvious but are out of scope. Name them. Kill them here.]

## Handoff
<!-- Arc:vision reads this section to generate docs/vision.md -->
purpose: [One sentence]
audience: [Behavior-based]
goals:
  - [Goal 1 (measurable)]
  - [Goal 2 (measurable)]
  - [Goal 3 (measurable)]
success: [Quantitative success criteria]
non_goals:
  - [Explicit exclusion 1]
  - [Explicit exclusion 2]
principles:
  - [Decision-making heuristic 1]
  - [Decision-making heuristic 2]
```

### Scoring Rubric: The Brief Quality Gate

Three layers. All three must pass. The brief is scored after writing and before
locking. If any layer fails, revise before moving to Step 5 (Spec).

---

#### Layer 1: Strategic Integrity (/50)

Does the strategy hold? Would it survive a smart person asking "why?" five times?

| Dimension | 10 | 7 | 4 | 1 |
|---|---|---|---|---|
| **Rallying Cry** | Memorable, makes a choice, competitor can't claim it. You'd put it on a wall. | Clear and directional but could be sharper. You'd put it in a deck. | Generic. Any company in the category could use it. | Missing, or a full sentence disguised as a rallying cry. |
| **Problem Specificity** | Named user, named behavior, named cost. Data from research. You feel the pain. | Real problem but could be more concrete. Some data, some abstraction. | Described at category level, not user level. "Users want better X." | Vague or missing. The problem is assumed, not stated. |
| **Proposition Clarity** | One paragraph, three sentences or fewer. A smart outsider gets it on first read. Different from competitors. | Clear but takes two reads. Or clear but not differentiated. | Requires context from the research to understand. | Jargon-laden, circular, or tries to be everything. |
| **Exclusions Quality** | Names things the client expects but we're cutting. Each exclusion saves real scope. Controversial ones included. | Has exclusions but they're safe picks nobody would argue with. | Generic ("out of scope: things not in scope"). | Missing or pro-forma. |
| **Durability** | You could hand this to a new team member in six months and they'd know exactly what to build and what not to build. | Mostly durable but some sections assume context from the kickoff. | Would need a walkthrough to understand. | Time-bound. Won't make sense in a month. |

**Threshold: 35/50. Below this, the strategy isn't locked. Revise.**

**Calibration examples:**
- Punt dossier headline "No one combines all four verticals. Betr tried. It's struggling." = 10/10 Problem Specificity. Named competitors, named the failure, named the opening.
- "We need to create a best-in-class digital experience" = 1/10 Proposition Clarity. Says nothing. Any company, any project.
- GamePass "More gaming" = 10/10 Rallying Cry. Two words. Makes a choice (breadth over depth). Microsoft's competitors were choosing "best games" or "exclusive games." GamePass chose volume.

---

#### Layer 2: Voice Quality (/50)

Does it sound like Julian wrote it? Apply Voice DNA scoring.

| Dimension | Question |
|---|---|
| Directness | Statements or announcements? |
| Rhythm | Varied or metronomic? |
| Trust | Respects reader intelligence? |
| Authenticity | Sounds human? |
| Density | Anything cuttable? |

**Threshold: 35/50.**

**Quick checks (from voice-dna.md):**
- Any adverbs? Kill them.
- Any passive voice? Find the actor.
- Any "here's what/this/that" throat-clearing? Cut to point.
- Any em dashes? Remove.
- Any vague declaratives? Name the specific thing.
- Any sentence where an inanimate object does a human verb? Name the person.

**Calibration:**
- "The gap between 'I have an opinion' and 'I'll put money on it' is friction, not motivation." = passes. Specific. Physical. Human subject implied.
- "In today's rapidly evolving digital landscape, brands need to leverage omnichannel strategies to create meaningful connections." = fatal. Every banned phrase in one sentence.

---

#### Layer 3: Polish (/50) — Client-facing briefs only

Is it clear enough that a smart person outside the industry follows it on first read?
Apply Copy Polish scoring.

| Dimension | Question |
|---|---|
| Accessibility | Could a smart non-expert follow this? |
| Jargon | Any undefined terms surviving? |
| Sentence load | Any sentence carrying more than one idea? |
| Evidence | Every claim backed by a specific number, name, or example? |
| Order | Information arranged for the reader's logic, not the writer's? |

**Threshold: 35/50.**

**Quick checks (from copy-polish.md):**
- Every acronym expanded on first use?
- Every industry term defined or replaced?
- Any claim without a specific number, name, or example?
- Lead sentence of each section states the point (not sets it up)?

**Calibration:**
- "The sweepstakes audience is a 39-year-old woman. Punt's target doesn't exist there." = passes. Plain language. Specific claim. No jargon.
- "The TAM for prediction markets is projected at $1.3T by 2030, with a 15-25% cross-product conversion rate driving iGaming ARPU uplift through ADW and DFS verticals." = fails. Acronym soup. Requires industry knowledge to parse.

## Research Dossier Scoring (Step 2)

The dossier is scored differently from the brief. It's an evidence document, not a strategy document. Calibrated against the Punt Research Dossier.

| Dimension | 10 | 7 | 4 | 1 |
|---|---|---|---|---|
| **Headlines as arguments** | Every section headline IS the claim. "No one combines all four verticals. Betr tried. It's struggling." | Most headlines are claims, some are labels. | Headlines are section labels. Arguments buried in body. | No headlines, or generic headers ("Market Overview"). |
| **Data density** | Every claim has a number, a name, or a date. Sources cited. "Kalshi grew from 600K to 5.1M users in one year." | Most claims sourced. Some assertions unsupported. | Some data, lots of assertions. | Opinion dressed as research. |
| **Assumption/Reality framing** | Key misconceptions identified and corrected with data. "The Assumption: X. The Reality: Y." Changes how the reader thinks. | Some assumptions challenged. Others accepted uncritically. | Research confirms what everyone already believes. | No original perspective. Category report energy. |
| **Competitive specificity** | Named competitors, named their weakness, named the opening. One line per competitor. Betr gets a red box because it's the closest analog. | Competitors named but analysis is surface-level. | "Competitive landscape" section with logos and no analysis. | Missing or generic. |
| **Tension extraction** | 2-3 real tradeoffs the strategy must resolve. Each one forces a choice. "Credibility vs. energy. The brand has to pick a leg to stand on." | Tensions identified but framed as problems, not choices. | Observations, not tensions. | No tensions. Just a market summary. |

**Threshold: 35/50. Below this, research isn't ready to feed the distill step.**
